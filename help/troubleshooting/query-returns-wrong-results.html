Query Returns Wrong Results

What this covers

Diagnosing incorrect values returned by Tessallite queries. Tessallite returns the same values whether a query hits an aggregate or the raw source — if the values are wrong, the cause is almost always in the model definition, data freshness, or a problem in the underlying source data, not in the routing layer.

Step 1 — Check the measure definition

Open Model Builder and navigate to the measure that is returning unexpected values.

  • Verify the source column is correct. A column rename in the source table produces wrong results silently if schema drift has not been detected.
  • Verify the aggregation type: SUM, COUNT, COUNT DISTINCT, AVG, MIN, MAX. An incorrect aggregation type is a common cause of wrong totals.
  • Check whether any filters are applied to the measure definition.

Step 2 — Check for grain mismatch

If a query asks for totals and an aggregate exists at a finer grain, Tessallite re-aggregates the pre-built summary to answer the query. This is correct for additive measures (SUM, MIN, MAX) but produces wrong results for COUNT DISTINCT.

If the query uses COUNT DISTINCT and returns a value that is too high, no exact-grain aggregate exists for that query pattern. To fix this:

  • Ask your Modeller to build an aggregate at the exact grain required by the query.
  • Or reformulate the query to use a grain for which an exact aggregate exists.

Step 3 — Check for stale data

An aggregate may not have been refreshed after the source data changed. In Model Builder, check the status column for each aggregate:

  • Ready: the aggregate is current.
  • Stale: the source has changed since the last build. Ask your Modeller to trigger a refresh.
  • Building: a refresh is in progress. Wait for it to complete.
  • Error: the last build failed. See Aggregates Not Building.

Step 4 — Check for schema drift

A source table column that a measure or dimension references may have changed type or been renamed since the model was built. Tessallite detects this and reports it in the Health tab.

  1. Open Model Builder.
  2. Select the project.
  3. Go to the Health tab.
  4. Look for "Schema drift detected" warnings. Each warning identifies the affected table and column.
  5. Update the model to reference the correct column name or type, then re-publish and rebuild aggregates.

Step 5 — Compare with a direct query

Run the equivalent query directly against the source data, bypassing Tessallite entirely. If the direct query also returns wrong values, the problem is in the source data — not in Tessallite. If the direct query returns correct values but Tessallite does not, return to Steps 1–4.

To run a direct query, connect to the source database using the source credentials (not the Tessallite JDBC endpoint) and execute the equivalent SQL.

Tessallite does not transform data. It routes queries and pre-aggregates, but the values it returns come from the source. If the source has data quality issues, Tessallite reflects them.
