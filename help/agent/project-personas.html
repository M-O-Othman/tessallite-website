Project-level personas

What this covers

Project-level personas restrict which models and attributes (measures and dimensions) the conversational agent can see and query within a project. They are distinct from model-level personas, which scope BI tool (JDBC/XMLA) access. This page explains the concept, the data model, the API, and how to use personas effectively.

Why project-level personas exist

A project may contain many models with hundreds of measures and dimensions. Not every audience needs access to everything. A finance team may need revenue and cost measures but not operational metrics. A regional manager may need only the models and attributes relevant to their territory. Without scoping, the agent sees the entire allow-listed model catalogue and may answer questions outside the intended audience.

Project-level personas solve this by defining named, reusable attribute scopes at the project level. Each persona declares which models it covers and, within each model, which measures and dimensions are visible. The agent's prompt assembly reads these scopes and filters the model profiles before the LLM sees them.

How personas differ from model-level personas

AspectModel-level persona (gateway)Project-level persona (agent)
PurposeColumn-level access control for JDBC/XMLA queriesAttribute scoping for the conversational agent

EnforcementGateway filters SELECT columns; returns 403 for restrictedAgent prompt omits restricted models and attributes
ConfigurationModel Builder > Row Security panelAgent persona API; chat toolbar picker

The two systems are independent. A project-level persona does not affect JDBC/XMLA queries, and a model-level persona does not affect the agent.

Data model

A project-level persona has a name, a slug (auto-generated from the name if not provided), and an optional description. Under each persona, zero or more model scopes define per-model attribute restrictions:

  • Model scope present: the persona can see that model. If included_measure_ids or included_dimension_ids are empty lists, the persona sees all attributes of that type. If non-empty, only the listed attributes are visible.
  • Model scope absent: the model is entirely excluded from the persona.

This is an allow-list model: only explicitly scoped models are visible, and within each model, only listed attributes (or all, if the list is empty).

Using personas in the agent chat

The persona picker appears in the agent chat toolbar, above the message area. It shows all personas defined for the project, plus a default option ("All models — no persona") that gives the agent access to the full allow-list.

Selecting a persona binds it to the current conversation. All subsequent questions in that conversation are scoped by the persona. Changing the persona on an existing conversation updates the scope for future turns; past turns retain their original scope.

When a persona is active:

  • The agent's system prompt only lists models and attributes allowed by the persona.
  • The LLM cannot reference excluded models or attributes because they are not in the prompt.
  • Query plans that somehow reference excluded objects are rejected by the validation layer.

Managing personas via the API

Persona CRUD is available at /api/v1/projects/{project_id}/agent/personas:

MethodPathDescription
GET/List all personas for the project
POST/Create a persona with optional model scopes
GET/{persona_id}Get a single persona
PATCH/{persona_id}Update name, description, or model scopes
DELETE/{persona_id}Delete a persona (conversations are unbound, not deleted)

Create example

POST /api/v1/projects/{project_id}/agent/personas
{
  "name": "Finance team",
  "description": "Revenue and cost measures only",
  "model_scopes": [
    {
      "model_id": "...",
      "included_measure_ids": ["revenue_id", "cost_id"],
      "included_dimension_ids": []
    }
  ]
}

An empty included_dimension_ids means the persona sees all dimensions in that model. Omitting a model from the scopes array excludes it entirely.

Design considerations

  • Start broad, then narrow. Begin with a persona that includes all models and restrict later. This avoids accidentally hiding data the audience needs.
  • One persona per audience. Avoid creating personas for individual users. Personas represent roles or teams.
  • Empty include lists = full access. Only populate include lists when you need to restrict specific attributes. If a model scope has no restrictions, leave both lists empty.
  • Persona deletion is safe. Deleting a persona sets persona_id to NULL on any bound conversations, reverting them to the default (all models) scope. No conversation data is lost.
