Parameterized Filters

What this covers

A parameterized filter is a named placeholder in a model’s query predicates that is resolved to a concrete value at query time. This page explains what parameters are, when to use them, how to define them, how values are resolved, and how BI tools pass parameter values through JDBC session variables.

What parameters solve

Many analytical models include filters that vary by user, session, or audience. Without parameters, the modeller must hard-code a default (inflexible), define separate models per filter value (duplication), or rely on downstream tools to append WHERE clauses (inconsistent). Parameters give the modeller a single declaration point for each variable filter.

Defining a parameter

  1. Open a model in Model Builder.
  2. Click the Parameters icon in the toolbelt.
  3. Click Add Parameter.
  4. Fill in: name (must start with @), display name, type, default value, allowed values, and description.
  5. Click Save.

Resolution order

When a query references a parameter, the value is resolved from three sources (highest precedence first):

  1. Persona default filter — if the query is bound to a persona that defines a filter for this parameter.
  2. JDBC session variable — set via SET app.<param_name> = <value> in the session.
  3. Model default value — the default defined on the parameter.

If no value is available from any source and the parameter has no default, the query fails with a clear error naming the unresolved parameter.

JDBC session variables

BI tools connected via JDBC can set parameter values per session:

SET app.region = 'APAC';
SET app.date_start = '2025-01-01';
SELECT * FROM sales WHERE region = @region;

The app. prefix is required. Session variables persist for the lifetime of the JDBC connection and are cleared on disconnect.

Worked example

A modeller defines @region (string, default "ALL") and @report_date (date_range). A DBeaver user runs SET app.region = 'EMEA' then queries — the engine resolves @region to 'EMEA' from the session variable. A different user connects via a persona with default_filters: {"region": "APAC"} and gets 'APAC' — the persona filter takes precedence.

Best practices

  • Use parameters for values that vary by audience. If every user sees the same filter, a regular WHERE clause is simpler.
  • Set sensible defaults. A parameter without a default requires every caller to supply a value.
  • Constrain with allowed values. For string and multi_value types, an allowed-values list prevents unexpected filter values.
  • Name consistently. Adopt a convention like @report_date_start, @region_filter.
