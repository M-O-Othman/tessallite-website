Service Not Starting

What this covers

Diagnosing a Tessallite service that fails to start, exits immediately, or restarts in a loop. This article applies to Docker Compose deployments. For other deployment targets, the diagnostic principles are the same, but the commands differ.

General diagnostic approach

  1. Check status: docker compose ps — look for any service showing Exit or Restarting status.
  2. Read logs: docker compose logs --tail=50 SERVICE_NAME — replace SERVICE_NAME with the name of the failing service (e.g., gateway, model-service).
  3. Fix the underlying cause: use the table below to identify the cause and apply the fix.
  4. Restart: docker compose restart SERVICE_NAME
  5. Verify: docker compose ps — confirm the service is Up.

To check the health endpoint after services are up:

curl http://localhost:3000/api/health

A response of {"status":"ok"} confirms the Frontend service is running and can reach the database.

Common causes by service

Service Symptom Likely cause Fix
postgres Exits immediately on start Data directory permissions are wrong or the volume was initialised with a different user docker compose down -v && docker compose up -d. Warning: the -v flag deletes all data volumes. Only use this if you are willing to lose existing data or are running a fresh install.
model-service Restarts in a loop Cannot connect to internal PostgreSQL — DB_HOST, DB_USER, or DB_PASS environment variables are wrong, or the postgres service has not finished initialising Verify DB_HOST, DB_USER, and DB_PASS in the .env file. Ensure the postgres service is healthy before starting the model-service: docker compose up -d postgres → wait for health → docker compose up -d model-service.
gateway Exits with "address already in use" Another process holds port 5433 or port 8080 Find and stop the conflicting process, or change GATEWAY_PORT_JDBC or GATEWAY_PORT_XMLA in the .env file to unused ports.
frontend Exits with "SESSION_SECRET not set" The SESSION_SECRET environment variable is missing from .env Add SESSION_SECRET=<a long random string> to the .env file and restart: docker compose restart frontend.
scheduler Restarts after a few minutes Cannot connect to the target schema or source database when attempting to build an aggregate Verify source connection parameters and target schema permissions. See Aggregates Not Building for permission grant commands.
Any service "Cannot connect to postgres" at startup The service started before the internal postgres service was ready — postgres takes a few seconds to initialise on first run Run docker compose restart SERVICE_NAME once postgres shows healthy status in docker compose ps.

Startup order and dependencies

Tessallite services depend on the internal PostgreSQL instance being fully initialised before they can connect. On first run, postgres may take 10–20 seconds to complete initialisation. If multiple services fail simultaneously after a fresh install, start postgres first and wait for it to be healthy before bringing up the remaining services:

docker compose up -d postgres
docker compose ps   # wait until postgres shows (healthy)
docker compose up -d

Service health is checked by the health endpoint at http://localhost:3000/api/health. If this endpoint returns {"status":"ok"}, all core services are running.
