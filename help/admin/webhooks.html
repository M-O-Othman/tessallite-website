Webhooks

What this covers

Tessallite can push notifications to external systems when significant events occur — a model is deployed, a user is created, or a setting changes. This page explains the webhook system: which events are available, how to create and manage endpoints, how to verify payload signatures, and how to handle delivery failures.

When to use webhooks

Webhooks are the right tool when an external system needs to react to something that happened inside Tessallite: CI/CD pipelines, audit systems, notification channels (Slack, Teams, PagerDuty), or data catalogs that sync model metadata.

Event types

EventFired when
model.publishedA model is deployed to a version.
model.undeployedA model’s deploy pointer is cleared.
model.deletedA model is permanently deleted.
user.createdA new user account is created.
user.deletedA user account is deleted.
settings.changedA tenant, project, or model setting is saved.

Creating an endpoint

  1. Navigate to Admin > Webhooks.
  2. Click New Endpoint.
  3. Enter a name, the destination URL, and select which event types to subscribe to.
  4. Click Save. Tessallite generates an HMAC-SHA256 signing secret.

Payload and signature

Each delivery is a POST with a JSON body containing event_type, timestamp, actor, and a detail object. The request includes an X-Tessallite-Signature header:

t=1746105000,v1=5d41402abc4b2a76b9719d911017c592...

Verify by computing HMAC-SHA256(signing_secret, t + "." + raw_body) and comparing to v1.

Maximum payload size is 64 KB.

Delivery and retry

AttemptDelay after failure
1stimmediate
2nd10 seconds
3rd60 seconds
4th (final)300 seconds

A 2xx response is success. After all retries are exhausted, the delivery moves to the dead-letter queue (DLQ).

Managing endpoints

  • Active toggle — disable an endpoint without deleting it.
  • Test delivery — send a synthetic webhook.test event.
  • Rotate secret — generate a new signing secret (immediate, requires confirmation).
  • Delivery history — per-endpoint log with status, response code, and timestamp.
  • DLQ tab — failed deliveries with retry and delete actions.

Best practices

  • Always verify signatures. Without verification, fake events can be sent to your endpoint.
  • Respond quickly. Return a 2xx within 5 seconds. Process asynchronously.
  • Use event filters. Subscribe only to the events you need.
  • Monitor the DLQ. A growing DLQ means your endpoint is failing.
  • Rotate secrets on team changes.
