Add Tables to a Model

What this covers

Before you can define dimensions, measures, or aggregates, you must add the source tables you intend to use into the model. This article explains how to navigate to the Model Builder, browse the schema, add tables, and choose the correct table type for each one.

Before you start

  • A project must exist and its data source connection must be working. See Create a Project and Add a Data Source.
  • You must be signed in as a Modeller or Tenant Admin on the project.

Navigating to the Model Builder

  1. Open the project from the workspace dashboard.
  2. Click Model Builder in the project navigation. The Model Builder opens with three areas:
    • Canvas — the central diagram showing tables and joins.
    • Toolbelt — the left sidebar with action buttons.
    • Drawer — the right panel showing properties of the selected object.

Adding a table

  1. In the Toolbelt, click Add Table. The Drawer opens showing the schema browser.
  2. In the schema browser, expand a schema to see the tables it contains. The browser lists all schemas and tables accessible to the data source user.
  3. Click a table name to select it. The Drawer shows the table's columns and a Table type selector.
  4. Choose the correct table type (see the reference below).
  5. Click Add to Model. The table appears as a node in the Canvas.
  6. Repeat for each table you need in the model.

Table type reference

Choosing the right type is important. The type determines how Tessallite constructs the aggregate grain — the combination of columns used when building pre-aggregated summary tables.

Table type When to use Example Included in aggregate grain?
fact The central transaction or event table. Every model must have exactly one fact table. This table drives the aggregate grain and typically contains numeric measures and foreign keys to dimension tables. orders, page_views, sensor_readings Yes — the fact table's key columns and measure columns form the base of every aggregate.
dim_aggregate A dimension table that participates in the aggregate grain. Use this type when you want the aggregate tables to be pre-grouped by columns in this dimension — for example, by date or by product category. dim_date, dim_product Yes — columns from this table can be selected as grouping keys in aggregates.
dim_detail A dimension table used only for filtering or labeling at query time. Its columns are not included in aggregate grain. Use this type for high-cardinality lookup tables where pre-aggregation at that level would be impractical. dim_customer, dim_employee No — queries that filter or group by columns from this table always fall through to raw data.

A model must have exactly one fact table. If you mark a second table as fact, the Health tab will show a validation error and the model cannot be published until the conflict is resolved.

What the Canvas shows after adding tables

Each table appears as a card in the Canvas. The card shows the table name and a type badge (fact, dim_aggregate, or dim_detail). Tables with no join connecting them to another table are shown with a dashed border, indicating they are disconnected. A disconnected table is valid while you are still building the model, but the Health tab will flag it as a warning until all tables are joined into the schema.

Click a table card in the Canvas to open its properties in the Drawer, where you can change its type or remove it from the model.

Removing a table

Select the table in the Canvas to open it in the Drawer, then click Remove from Model. All joins, dimensions, and measures that reference columns in that table are also removed. This action cannot be undone — you must re-add the table and redefine any dependent objects if you remove a table by mistake.
