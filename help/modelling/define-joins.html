Define Joins

What this covers

Joins tell Tessallite how tables relate to one another. Without joins, the query router cannot construct the SQL needed to pull dimension attributes into aggregate GROUP BY queries. This article covers the join creation flow, join properties, when to choose INNER vs LEFT, structural constraints, and how to edit or delete a join.

Before you start

  • All tables involved in a join must already be added to the model. See Add Tables to a Model.
  • You must know the foreign key relationship: which column in one table corresponds to the primary key in the other.

Creating a join

  1. Open the Model Builder for the project.
  2. In the Toolbelt, click Add Join. Alternatively, drag from one table card to another in the Canvas — the Drawer opens with the join form pre-populated with the two tables.
  3. In the Drawer, set the Left table and Left column. The left table is typically the fact table or the table on the many-side of the relationship.
  4. Set the Right table and Right column. The right table is typically the dimension table on the one-side.
  5. Choose the Join type: LEFT or INNER.
  6. Click Save Join. A line appears in the Canvas connecting the two table cards, labeled with the join type.

Join properties

Property Description
Left table The table on the left side of the ON clause. Tessallite generates LEFT_TABLE.left_column = RIGHT_TABLE.right_column in the query.
Left column The foreign key column in the left table.
Right table The table being joined to. For star schemas this is typically a dimension table.
Right column The primary key or join key column in the right table.
Join type LEFT or INNER. Controls how unmatched rows are handled at query time.

Choosing INNER vs LEFT

Use LEFT in almost all cases. A LEFT join preserves every row from the left (fact) table even when the right (dimension) table has no matching row. This prevents fact rows from being silently dropped from query results if a dimension record is missing or the foreign key is null.

Use INNER only when you are certain that every fact row has a matching dimension row and you want unmatched fact rows excluded. A misapplied INNER join is a common source of silent data loss — the query returns no error but produces totals lower than expected.

Every dimension table must be reachable from the fact table through a join path. A dimension table with no join connection to the fact table (directly or through another table) cannot be used in aggregate GROUP BY queries and will trigger a warning in the Health tab.

Structural constraints

Tessallite enforces the following constraints on the join graph:

  • Star or snowflake only. Joins must radiate outward from the fact table. Each dimension table connects to the fact table directly (star schema) or through another dimension table (snowflake schema). Cycles are not allowed.
  • No fact-to-fact joins. A join between two tables marked as fact is not supported. If you need data from two fact tables, use two separate projects or pre-join them in the source database before modelling.
  • One join per table pair. Only one join can exist between any two tables. If the relationship involves a composite key, define the join on the most selective column and handle the remaining key columns upstream in the source.

The Health tab shows errors for any constraint violations. The model cannot be published while errors are present.

Editing a join

Click the join line in the Canvas to open it in the Drawer. Edit any property and click Save Join to apply the change.

Deleting a join

Click the join line in the Canvas to open it in the Drawer, then click Delete Join. Deleting a join disconnects the two tables. Dimensions or measures that rely on columns in the disconnected table will produce errors in the Health tab until the join is restored or those objects are removed.
