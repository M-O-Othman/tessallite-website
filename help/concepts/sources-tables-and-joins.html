Sources, Tables, and Joins

What this covers

This article explains what a source is in Tessallite, how tables are classified within a model, how joins connect them, and why these classifications directly affect query routing and aggregate building.

Sources

A source is the data warehouse or query engine that a project connects to. Tessallite supports three source types: PostgreSQL, Google BigQuery, and Hadoop/Spark Thrift Server. The source connection is configured at the workspace level and is shared by all models in that workspace.

When you open Model Builder and add a table to a model, you are selecting a table or view that exists in the connected source. Tessallite reads the schema from the source to populate column lists. The source itself is never modified — Tessallite only reads from it. Pre-computed aggregate tables are written to a separate target schema, which may be in the same source or a different one.

The three table types

Every table added to a model is assigned one of three types. The type tells Tessallite how the table participates in aggregation and query routing.

Type Role Participates in aggregate grain Typical example
fact The primary transaction table. Drives the aggregate grain. Each model must have exactly one fact table. Yes — the fact table is the central table of every aggregate. orders, sales_events, page_views
dim_aggregate A dimension table whose columns may extend the aggregate grain. Joins to the fact table and its columns can be included in a grain definition. Yes — dimensions drawn from a dim_aggregate table extend the grain and are included in the pre-computed summary. dim_product, dim_date, dim_store
dim_detail A dimension table used for filtering and labelling only. Its columns can be used in WHERE clauses and display names, but they cannot define an aggregate grain. No — a query that groups by a dim_detail column cannot be served by a pre-computed aggregate and will fall through to raw data. dim_customer_contact, dim_employee_notes

Use dim_detail sparingly. Any dimension drawn from a dim_detail table bypasses aggregates entirely. If a column is frequently used for grouping, its source table should be classified as dim_aggregate.

The fact table as primary table

Every model must have exactly one fact table. This is the central table from which all aggregates are built. Every join in the model connects either directly or transitively back to the fact table. Tessallite uses the fact table as the anchor when generating the SQL that builds aggregate summary tables.

If a model contains no fact table, the Model Builder will surface a "Fact table missing" error in the Health tab and no aggregates can be built.

Joins

A join connects two tables within a model. Each join is defined by:

  • Left table and left column — the column on the driving side of the join.
  • Right table and right column — the column being joined to.
  • Join type — either LEFT or INNER.

Use INNER when every row in the fact table is guaranteed to have a matching row in the dimension table. Use LEFT when the dimension is optional — rows without a match will still appear in query results, with NULL dimension values.

A join is expressed in the aggregate build SQL as:

LEFT JOIN dim_product
  ON orders.product_id = dim_product.product_id

Joins are not transitive by default. If you join orders to dim_product, and you want to access a column from dim_brand, you must define a separate join from dim_product to dim_brand.

If a column named in a join definition is later removed from the source schema, Tessallite will detect the discrepancy during its next schema sync and raise a "Join column not found" error in the Health tab. The affected joins must be updated or removed before aggregate builds will resume.

Why table type matters for routing

When the Query Router evaluates an incoming query, it checks whether a pre-computed aggregate can satisfy it. Part of that check is whether every GROUP BY column in the query is drawn from a fact or dim_aggregate table. A column from a dim_detail table is not tracked in any aggregate grain, so the router cannot confirm a match and the query falls through to raw data.

This means that classifying a frequently-grouped column's source table as dim_detail instead of dim_aggregate silently prevents aggregate acceleration for all queries that include that column.
