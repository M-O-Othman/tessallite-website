Workspace Settings

What this covers

The Settings tab in the Admin panel controls workspace-level configuration: display name, aggregate scheduling defaults, query timeout, schema sync schedule, and session timeout. This article lists each setting, its default, and its effect. It also covers how to delete a workspace.

Navigating to workspace settings

  1. In the workspace sidebar, click Admin.
  2. Click the Settings tab.

Settings reference

Setting Description Default Notes
Display name The workspace label shown in the UI. Visible to all users in the workspace. Set at creation Can be changed at any time. Does not affect the slug or any connection strings.
Default aggregate schedule The cron expression applied automatically to new aggregates unless an explicit schedule is set in the aggregate Drawer. 0 2 * * * (daily at 2:00 AM) Changing this value affects only aggregates created after the change. Existing aggregates keep their current schedule.
Query timeout Seconds before a raw-source query (one that cannot be answered from an aggregate) is cancelled and an error is returned to the client. 300 Applies to new query connections. Queries already in flight are not affected until they are re-submitted.
Schema sync schedule Cron expression for automatic schema sync, which re-reads source table column definitions and flags any schema drift in the Health tab. 0 1 * * * (daily at 1:00 AM) Schema sync does not modify the model automatically. It only detects drift and reports it as a Warning diagnostic.
Session timeout Minutes of inactivity before a user is signed out automatically. 60 Takes effect for new sessions after saving. Existing sessions use the timeout value that was in effect when the session was established.

Saving settings

  1. Make the required changes on the Settings tab.
  2. Click Save Settings.

Changes apply immediately for new connections and new sessions. Active sessions continue under the settings that were in effect when they authenticated. If you need a changed timeout to apply to all users immediately, ask them to sign out and sign back in.

Deleting the workspace

The Settings tab includes a Danger Zone section at the bottom of the page. Workspace deletion is accessed from there.

Workspace deletion is irreversible. Deleting the workspace permanently removes all projects, models, aggregate metadata, and user records associated with it. Source data in your connected databases is not affected. There is no undo.

  1. In the Danger Zone section, click Delete Workspace.
  2. A confirmation dialog appears. Type the workspace slug exactly as shown to confirm.
  3. Click Confirm Delete.

The workspace is removed immediately. All users lose access. The System Admin will see the workspace removed from the Workspaces tab in the System Administration screen.
