Add a Data Source

What this covers

Each Tessallite project connects to exactly one data source. This article explains how to view or change the data source connection from the project Settings panel, describes the connection parameters for each supported source type, and lists common connection failure causes and remedies.

Navigating to the data source settings

  1. Open the project from the workspace dashboard.
  2. Click the Settings tab at the top of the project page.
  3. Locate the Data Source section. The current source type and connection parameters are displayed here.
  4. To change the source type, select a new type from the Source type dropdown. Fill in the parameters for the new type.
  5. Click Test Connection to verify the new parameters before saving.
  6. Click Save to apply the change.

Changing the source type after tables have been added to the model will invalidate all table references. The model's Canvas will show errors on every table. You must remove all existing tables and re-add them from the new source before the model can be used.

Connection parameter reference

Source type Parameter Required Notes
PostgreSQL Host Yes Hostname or IP of the PostgreSQL server. For Cloud SQL on GCP, use the private IP or Cloud SQL Auth Proxy address.
Port Yes Default is 5432.
Database Yes The database name containing your schemas and tables.
User Yes Database user with SELECT access on the target schemas.
Password Yes Stored encrypted; never shown in plain text after saving.
BigQuery Project ID Yes The GCP project ID that owns the BigQuery dataset.
Dataset Yes The BigQuery dataset to expose. A project can only connect to one dataset at a time.
Service account JSON Yes Paste the full JSON key file content. The service account must have bigquery.dataViewer and bigquery.jobUser roles.
Hadoop / Spark Thrift Server Thrift Server host Yes Hostname or IP of the HiveServer2 or Spark Thrift Server endpoint.
Port Yes Default is 10000 for HiveServer2 and 10001 for Spark Thrift Server.
Database Yes The Hive or Spark database (schema) name.

What the Test Connection button checks

Clicking Test Connection asks the Tessallite gateway to attempt an authenticated connection to the source using the parameters you entered. It verifies:

  • The host is reachable from the gateway's network.
  • The port is open and the database service is listening.
  • The credentials are accepted by the source.
  • For BigQuery: the service account JSON is valid and the account has sufficient permissions.

A successful test confirms the gateway can authenticate. It does not run queries or validate that specific schemas or tables exist.

Common connection failures

Error Likely cause Remedy
Connection timed out The host is not reachable from the gateway due to a network or firewall block. Check that the gateway's outbound IP is allowed through the source's firewall or VPC rules.
Connection refused The database service is not running on the specified port, or the port is wrong. Verify the port number. Confirm the database service is running.
Authentication failed Wrong username or password (PostgreSQL/Hadoop), or invalid service account key (BigQuery). Re-enter credentials. For BigQuery, generate a new key in the GCP console and paste the updated JSON.
Database does not exist The database name is misspelled or the user does not have access to it. Confirm the database name and that the user has been granted access.
Permission denied The user can connect but lacks SELECT permission on the target schema. Grant SELECT on the relevant schemas and tables in the source database.
