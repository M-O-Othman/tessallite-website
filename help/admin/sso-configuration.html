SSO Configuration

What this covers

Tessallite supports SAML 2.0 and OpenID Connect (OIDC) for single sign-on. This page explains how SSO works, how to configure each protocol, how to set up group-to-role mappings, and what happens when a user signs in via SSO for the first time.

How SSO fits into the authentication chain

Tessallite authenticates users through a chain of backends. Each tenant can enable one or more of: local, ldap, gcp_iam, saml, and oidc. When saml or oidc is enabled, a Sign in with SSO button appears on the login page alongside the standard credential form.

SAML and OIDC are redirect-based. The user’s browser is sent to the identity provider (IdP) for authentication. After successful authentication, the browser is redirected back to Tessallite, which validates the response, creates or updates the local user record, and issues a JWT.

SAML 2.0 setup

Step 1 — Register Tessallite as a Service Provider

Download the SP metadata from GET /api/v1/auth/saml/metadata and upload it to your IdP (Auth0, Google Workspace, Okta, AD FS, or any SAML 2.0 IdP).

Step 2 — Configure the IdP in Tessallite

SettingPurpose
SAML_IDP_METADATA_URLURL to the IdP’s SAML metadata (auto-fetched). Use this OR the XML field.
SAML_IDP_METADATA_XMLPaste the IdP metadata XML directly when the URL is not accessible.
SAML_SP_ENTITY_IDThe entity ID Tessallite uses to identify itself. Must match the IdP registration.

Step 3 — Map attributes

SettingDefaultWhat it maps to
SAML_ATTR_EMAILNameIDThe user’s email address.
SAML_ATTR_DISPLAY_NAME(none)The user’s display name.
SAML_ATTR_GROUPS(none)Group memberships for role mapping.

OIDC setup

Step 1 — Register Tessallite with the IdP

Create an OAuth 2.0 / OIDC client in your IdP. Set the redirect URI to GET /api/v1/auth/oidc/callback.

Step 2 — Configure the IdP in Tessallite

SettingPurpose
OIDC_ISSUERThe issuer URL. Tessallite fetches .well-known/openid-configuration from here.
OIDC_CLIENT_IDThe client ID from the IdP registration.
OIDC_CLIENT_SECRETThe client secret (Fernet-encrypted at rest).
OIDC_SCOPESOAuth scopes to request (default: openid email profile).
OIDC_GROUPS_CLAIMClaim name containing group memberships (default: groups).

Group-to-role mapping

Both SSO backends extract group memberships from the IdP response. Group-to-role mappings tell Tessallite which role to assign for each IdP group.

  1. On each SSO login, Tessallite reads the user’s groups from the IdP response.
  2. It looks up every group name in the idp_group_role_mappings table.
  3. For each match, it creates or updates a UserAccessBinding entry with the mapped role and project scope.
  4. If none of the user’s groups match any mapping, the user inherits the JIT default role.

Manage mappings in Admin > Settings > Group Mappings. Each mapping has three fields: IdP group name (case-sensitive), project (optional), and role (admin, modeler, or viewer).

JIT user creation

When a user authenticates via SSO for the first time and no local user record exists, Tessallite automatically creates one. The auth_source is set to saml or oidc. Users with a non-local auth source cannot change their password via Tessallite.

Best practices

  • Test with one user first. Enable SSO, sign in yourself, and verify the attribute mapping before rolling out.
  • Keep local auth enabled as a fallback. Until SSO is stable, keep local in AUTH_BACKENDS.
  • Set the groups attribute/claim early. Group-to-role mapping only works when the IdP sends group information.
  • Use tenant-wide mappings sparingly. Prefer project-scoped mappings for least-privilege.

Troubleshooting

  • “Sign in with SSO” button does not appear. Check that saml or oidc is listed in AUTH_BACKENDS and that the IdP configuration fields are non-empty.
  • “Invalid assertion” error. The SP entity ID does not match the IdP registration, or the ACS URL is wrong.
  • User created but has no project access. The user’s IdP groups do not match any mapping, and the JIT default role has no project bindings.
