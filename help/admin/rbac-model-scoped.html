Model-Scoped RBAC

What this covers

Model-scoped RBAC lets you grant a user a different access level on a specific model than they have at the project level.

The binding hierarchy

Access is determined by checking bindings in this order:

  1. Model-scoped binding — takes precedence for operations on that specific model.
  2. Project-scoped binding — used when no model-scoped binding exists.
  3. Bootstrap rule — a tenant_admin user can access any model even with no explicit binding.

If none match, the request is rejected with HTTP 403.

Why model-scoped bindings exist

The typical use case is a project with multiple models where different teams own different models:

  • Data Engineering needs modeler rights on the raw-data model but viewer only on the marketing attribution model.
  • An external partner can be granted viewer on one model without access to the rest of the project.
  • A model under active development can be scoped to modeler-only access while other models remain available to viewers.

Creating a model-scoped binding

  1. Open Tenant Administration › Projects › [your project] › Access.
  2. Locate the user row and click the model-scope icon.
  3. Select the target model and the role to grant.
  4. Save.

Or via API:

POST /api/v1/projects/{project_id}/access
{
  "user_id": "<uuid>",
  "role": "modeler",
  "model_id": "<uuid>"
}

Revoking a binding

DELETE /api/v1/projects/{project_id}/access/{binding_id}

The user retains their project-level binding for other models.

Interaction with row security and personas

Model-scoped bindings control which CRUD operations a user can perform on model metadata. Row security and persona restrictions apply independently and are not affected by the binding level.

Roles reference

RoleWhat it permits
viewerRead metadata, run queries, view diagnostics
modelerViewer rights plus create/edit/delete dimensions, measures, aggregates, rules
adminModeler rights plus manage connections, targets, deploy/undeploy
