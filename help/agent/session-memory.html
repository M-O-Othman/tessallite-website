Agent Session Memory

What this covers

The conversational agent maintains a history of turns (question-answer pairs) within each conversation. This page explains how session memory works, how the history depth setting affects prompt construction, and how to manage conversation data through stats and purge controls.

How session memory works

Each conversation is stored as a sequence of turns. A turn consists of the user’s question and the agent’s response (answer text, citations, thought process, and metadata). When a new question arrives, the agent includes prior turns in the LLM prompt so the model has conversational context — handling follow-ups, resolving pronouns, and maintaining coherence.

More history means a larger prompt, which consumes more tokens and increases latency. The session history depth setting controls this trade-off.

Session history depth

The session_history_depth setting (default: 20) controls how many recent turns include the full agent response in the prompt. Older turns appear in a compressed form: only the user’s question is included, prefixed with [earlier question].

For a conversation with 30 turns and a depth of 20: turns 1–10 show only the user’s question; turns 11–30 show both the question and the agent’s response.

Choosing a depth

ScenarioSuggested depth
Short, focused Q&A sessions10–20
Long analytical conversations30–50
High-throughput API usage5–10
Models with very long responses5–15

Configuring

  1. Open Tenant Administration > Projects.
  2. Click the gear icon on the project row.
  3. Navigate to the Advanced tab under agent settings.
  4. Set the Session history depth value (range: 1–100).
  5. Click Save.

Conversation stats

The Advanced tab displays the total number of conversations and the date of the oldest conversation for the project.

Purging conversations

Purge by age

Enter a number of days and click Purge. Every conversation whose last activity is older than the specified number of days is permanently deleted.

Delete all

Click Delete All Conversations and type DELETE ALL to confirm. Every conversation for the project is permanently removed.

Both actions are hard deletes — the data cannot be recovered.

Relationship to retention

MechanismTriggerGranularity
Retention settingAutomatic (daily scheduler job)Conversations older than configured days
Manual purgeOn-demand (admin clicks button)By age threshold or all

Best practices

  • Set retention for ongoing cleanup. Manual purges are for one-off situations.
  • Start with default depth. 20 turns covers most conversational patterns.
  • Purge test data before production launch. Test conversations skew stats and waste storage.
