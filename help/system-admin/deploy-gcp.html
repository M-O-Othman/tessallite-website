Deploy on GCP

What this covers

GCP-specific architecture for a production Tessallite deployment: Cloud Run services, Cloud SQL, load balancers, service account permissions, health check configuration, and instance scaling recommendations.

Architecture on GCP

Each Tessallite service runs as a separate Cloud Run service. The internal PostgreSQL database is replaced by a Cloud SQL for PostgreSQL instance. Two load balancers handle external traffic:

  • An HTTPS Load Balancer terminates SSL and routes port 8080 (XMLA) and port 3000 (frontend) traffic to the corresponding Cloud Run services.
  • A TCP Load Balancer (or proxy arrangement — see below) handles port 5433 JDBC traffic for gateway.

JDBC and port 5433 on Cloud Run

Cloud Run natively serves HTTP and HTTPS. It does not natively accept raw TCP connections on arbitrary ports such as 5433. Two options exist for JDBC access:

  • Option A (recommended for production): Deploy gateway to a Compute Engine VM or GKE node pool instead of Cloud Run. This gives a stable TCP endpoint on port 5433 with no protocol translation overhead.
  • Option B (experimental): Use Cloud Run with a TCP proxy forwarding rule. This is not supported in all regions and carries no SLA from Google. Verify regional availability before adopting.

The XMLA endpoint (port 8080) does not have this limitation and runs on Cloud Run without modification.

Environment variable differences from local

On GCP, the following variables differ from a local Docker Compose deployment:

  • DB_HOST: Set to the Cloud SQL private IP address, or to 127.0.0.1 if using the Cloud SQL Auth Proxy sidecar.
  • SESSION_SECRET: Must be set. Store it in GCP Secret Manager and mount it as an environment variable in Cloud Run — do not embed it in the service definition.
  • DB_PASS: Store in GCP Secret Manager. Mount via the Variables and Secrets tab in Cloud Run.

Cloud SQL connection

Two methods for connecting Cloud Run services to Cloud SQL:

  • Cloud SQL Auth Proxy sidecar: Add the proxy as a sidecar container in the Cloud Run service. Set DB_HOST=127.0.0.1. The proxy handles authentication and encryption.
  • Private IP: Connect Cloud Run to a VPC using Direct VPC Egress or Serverless VPC Access. Set DB_HOST to the Cloud SQL private IP. Requires the Cloud SQL instance to have a private IP assigned.

Service account permissions

Create a dedicated service account for Tessallite services. Grant the following IAM roles:

Role Required by Purpose
roles/cloudsql.client All services except gateway Authenticate to Cloud SQL
roles/run.invoker All services Allow inter-service HTTP calls within Cloud Run
roles/artifactregistry.reader All services Pull container images from Artifact Registry (if using private registry)
roles/secretmanager.secretAccessor frontend, all services needing secrets Read secrets from Secret Manager at startup

Health checks

Each service exposes a health endpoint at GET /api/health on its HTTP port. Cloud Run readiness probes should target this path. A healthy response returns HTTP 200 with body {"status":"ok"}. Configure the probe with an initial delay of 10 seconds and a period of 30 seconds.

Minimum instances

Cloud Run scales to zero by default, which causes cold starts. Recommended minimums:

  • gateway: Set --min-instances=1. JDBC clients hold long-lived connections and cannot tolerate cold-start delays.
  • All other services: Can run at minimum 0 for cost savings. Cold starts on internal HTTP calls are acceptable in most workloads.

Estimated cost

For light usage (development or small team): Cloud SQL db-f1-micro plus Cloud Run on-demand pricing runs approximately $30–60 per month. Cost scales primarily with Cloud SQL instance size and query volume routed through gateway.
