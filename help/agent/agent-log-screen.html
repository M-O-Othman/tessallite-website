Agent log screen

What this covers

The agent log screen is a full-trace audit viewer for every conversational agent turn in a project. It lets tenant administrators and project modellers inspect what happened during any agent interaction: the user's question, the LLM's reasoning, the queries it chose, the answers it produced, how the judge evaluated those answers, and how tokens were spent. This article explains the purpose of the log screen, how to enable it, how to read the trace data, and how to use it effectively for quality assurance and troubleshooting.

Why the log screen exists

The conversational agent sits between users and the data layer. When an answer is wrong, slow, or refused, someone needs to understand why. The chat interface shows the outcome, but not the machinery behind it. The log screen exposes that machinery: every decision the agent made, every query it ran, every judgement it received.

Common use cases: answer quality review after enabling the agent for a new team, judge calibration by examining fail and warn verdicts, token cost investigation after a usage spike, and user support when a specific question produced a surprising answer.

Enabling the log screen

The log screen is controlled by a project-level toggle. Open Tenant Administration, select the project, click the gear icon, and navigate to Setup under the Conversational Agent section. The main agent toggle must be on. Below it, switch Agent log screen enabled on and click Save.

After saving and closing the configuration drawer, a history icon appears in the project header bar next to the chat icon. Clicking it opens the log screen in a new browser tab. Disabling the toggle hides the entry point but does not delete any trace data.

Reading the log table

The table shows one row per agent turn, ordered by conversation (most recently active first) and then by turn position within each conversation. Summary columns include timestamp, user, question (truncated), status (ok, refused, error), judge verdict (pass, warn, fail), query route, result row count, input/output token counts, and latency.

Click any row to expand it. The expanded view splits into two columns:

  • Left column — the agent's work: prompt sent to the LLM (the full system prompt and user message payload), the raw LLM response (unprocessed text before tool-call parsing), thought process, LLM plan, semantic query, physical SQL, and the prose answer.
  • Right column — evaluation and feedback: citations, judge reasoning, judge metrics, guardrail actions, and user feedback (thumbs-up/down and comments).

Using the filters

The filter bar at the top narrows the table. From/To sets a date range. User filters by caller identity. Question contains does a substring search on the user's message. Click Search to apply; the record count updates. Filters combine with AND logic, and pagination resets to page one after each search.

Best practices

  • Review early and often. The first days after enabling the agent for a new audience reveal misinterpretation patterns quickly — wrong dimension mappings, aggregate grain mismatches, ambiguous business terms.
  • Triage by judge verdict. Fail verdicts need investigation. Warn verdicts need a judgment call on rubric strictness. Pass verdicts still deserve spot checks for rubric blind spots.
  • Compare semantic query to physical SQL. When the answer is wrong, the gap between intent and execution is usually visible here. A correct semantic query with wrong SQL points to the rewriting layer or a stale aggregate. A wrong semantic query points to the LLM's interpretation — check the glossary, model context, or example questions.
  • Watch token usage. High token counts on simple questions may indicate verbose model context, LLM over-reasoning, or excessive judge output.
  • Combine with the judge rubric. The log screen shows what the judge scored; the rubric controls how it scores. Consistent mismatches between human judgement and judge verdicts signal that the rubric needs revision.
