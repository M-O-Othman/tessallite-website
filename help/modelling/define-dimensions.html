Define Dimensions

What this covers

A dimension is a named grouping attribute that maps a business-friendly label to a specific source column. Dimensions appear as columns in the virtual schema that BI tools query. This article explains dimension fields, the definition flow, naming conventions, and what happens when a dimension references a column that no longer exists.

Before you start

  • All tables that contain the columns you want to use as dimensions must be added to the model. See Add Tables to a Model.
  • Tables that are not the fact table must be connected to the fact table by a join path. See Define Joins.

What a dimension is

A dimension maps a business name to a column in the model's source tables. For example, the database column orders.status_cd might be exposed as a dimension named Order Status. BI tools see Order Status as a queryable column in the model's virtual schema.

Dimensions from any table in the model — fact, dim_aggregate, or dim_detail — are all valid. The difference is how they behave in aggregates: dimensions from dim_aggregate tables can be included in the aggregate grain, while dimensions from dim_detail tables always cause a query to fall through to raw data.

Dimension fields

Field Required Description
Name Yes The identifier for this dimension in the model's schema. Must be unique within the model. Use a business-friendly name in title case. BI tools display this name to analysts.
Source table Yes The table in the model that contains the column. Any table added to the model is available here.
Source column Yes The column in the source table. Only columns in the selected source table are shown.
Display name No An alternate label shown in BI tool interfaces that support display names. If left blank, the Name is used.
Description No Free-text explanation of what this dimension represents. Shown in the model's metadata and in some BI tools as a tooltip.
Default sort order No ASC or DESC. Controls the default sort direction when an analyst orders by this dimension. Does not affect aggregate construction.

Defining a dimension

  1. Open the Model Builder for the project.
  2. Select a table in the Canvas, or click Add Dimension in the Toolbelt. The Drawer opens showing the dimension form.
  3. If you used the Toolbelt button, select the Source table from the dropdown first.
  4. Select the Source column from the list. Only columns in the selected source table are shown.
  5. Enter a Name. Use a business-friendly name rather than the raw column name — for example, use Sale Date rather than sale_dt.
  6. Optionally fill in the Display name, Description, and Default sort order.
  7. Click Save. The dimension count in the Summary Bar at the bottom of the Model Builder increments by one.

Example dimensions

Dimension name Source table Source column Notes
Sale Date dim_date date_actual From a dim_aggregate table; can be used as an aggregate grouping key.
Product Category dim_product category_name From a dim_aggregate table; enables pre-aggregation by category.
Order Status orders status_cd From the fact table itself; valid as a dimension for filtering or grouping.
Customer Name dim_customer full_name From a dim_detail table; queries using this dimension fall through to raw data.

Naming conventions

  • Use business terminology, not database column names. Analysts see dimension names directly in their BI tools.
  • Use title case: Order Status, not order_status or ORDER_STATUS.
  • Avoid abbreviations that are not universally understood within your organization.
  • Dimension names must be unique within the model. If two tables have columns that represent the same concept, choose one as the canonical source or differentiate the name (for example, Ship Date vs Order Date).

How dimensions appear to BI tools

When a model is published, each dimension appears as a column in the model's virtual schema. BI tools connecting over JDBC see the workspace slug as the database name, the model name as the schema name, and each dimension and measure as a column in a virtual table. Analysts can GROUP BY, filter on, or ORDER BY any dimension column.

If a dimension references a missing column

If the source table's schema changes after a dimension is defined — for example, the underlying column is renamed or dropped in the source database — the dimension becomes invalid. The Health tab in the Model Builder shows an error identifying the affected dimension and the missing column. The model cannot be published while errors are present. Resolve the error by editing the dimension to point to the correct column, or delete the dimension if the column has been removed permanently.
