Architecture Overview

What this covers

The six services that make up Tessallite, the internal database, how data flows through the platform, and where aggregates are physically stored.

Services

gateway

The public entry point for query traffic. Exposes two protocol endpoints: a PostgreSQL wire protocol listener on port 5433 for JDBC clients, and an XMLA listener on port 8080 for Excel and Power BI. Gateway translates incoming queries into an internal format and forwards them to query-router. It holds no query state between requests.

query-router

Receives translated queries from gateway. For each query, query-router asks model-service whether a pre-built aggregate can satisfy it. If a matching aggregate exists and is current, query-router rewrites the query to target that aggregate table in the user's source. Otherwise the original query goes directly to the raw source tables. Query-router records a miss in the model-service log whenever it falls back to the source.

model-service

The semantic layer. Stores the definitions of tables, joins, dimensions, measures, and aggregates. When query-router asks whether an aggregate can answer a query, model-service evaluates the request against registered definitions and responds with a routing decision. All model data is persisted in the internal PostgreSQL database.

optimizer

Reads the query miss log written by model-service. A miss is recorded each time query-router fell back to the raw data source because no aggregate matched. Optimizer scores candidate aggregates by return on investment — how many queries they would accelerate relative to the cost of building and refreshing them. Candidates exceeding the configured score threshold are surfaced as build recommendations in the frontend.

scheduler

Builds and refreshes aggregate tables in the user's target schema on their data source. Runs on a configurable cron schedule and can be triggered on demand from the frontend. Scheduler reads aggregate definitions from model-service, executes the build queries against the source, and records completion status in the internal database. It never writes user data into Tessallite's internal database.

frontend

Web interface served on port 3000. Used by System Admins, Workspace Admins, and Data Analysts for model building, aggregate management, connection setup, and platform monitoring. Communicates with model-service and optimizer via internal HTTP calls.

Internal PostgreSQL database

Tessallite runs its own PostgreSQL instance (port 5432, internal only) to store operational state. This database holds workspace metadata, semantic model definitions, aggregate build history, and the query miss log. It does not store any of the user's source data or the content of aggregate tables — those live in the user's own data source.

The internal PostgreSQL database is not exposed outside the container network. No BI tool or external application should connect to it directly.

Data flow

  1. A BI tool sends a query to gateway via JDBC (port 5433) or XMLA (port 8080).
  2. Gateway translates the query into an internal format and forwards it to query-router.
  3. Query-router asks model-service whether any registered aggregate satisfies the query.
  4. If an aggregate matches: query-router rewrites the query to target the aggregate table in the user's source, executes it, and returns results to the BI tool via gateway.
  5. If no aggregate matches: query-router routes the query to the raw source tables, records a miss in the model-service log, and returns results via gateway.

Where aggregates live

Aggregate tables are written to the user's own data source — in the target schema or dataset specified when configuring the connection. They are not stored in Tessallite's internal PostgreSQL. Tessallite only stores the definition and build status of each aggregate. You can inspect or query aggregate tables directly in your data warehouse alongside source tables.

Stateless service design

All six services keep no local state on disk. All persistent state lives in the internal PostgreSQL database. Any service can be stopped, restarted, or replaced at any time without data loss. Services can be scaled horizontally by running multiple instances behind a load balancer; the internal PostgreSQL remains a single instance and is the authoritative state store.
