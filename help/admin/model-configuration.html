Model Configuration

What this covers

The Configuration sub-tab inside the Model Builder Settings panel lets you override per-model knobs that only make sense for an individual model. It is laid out as five focused tabs — Aggregates, AI Optimizer, Pocket, Predictive, Limits — so a modeller can tune the parts that matter without scanning unrelated settings.

Where to find it

  1. Open the Model Builder for the model you want to configure.
  2. Click the Settings icon in the toolbelt (right side).
  3. Switch to the Configuration sub-tab. It is only available while a model is open.

How overrides work

Every row is a pure override. The placeholder shows the value currently inherited from the project level, which itself may fall back to system defaults. Save a value to override; leave the field blank and click Save, or click Clear, to remove the override and inherit again. The resolver walks model → project → system → registry default and returns the first non-null value.

The five tabs

Aggregates

Per-model defaults for the materialised aggregate pipeline:

  • Default aggregate cron — applied to a new aggregate on this model when no explicit schedule is set.
  • Refresh window and timeout — bounds the scheduler's per-aggregate run.

Override these when a single model has unusually heavy aggregates that need a slower cadence than the project's other models.

AI Optimizer

Knobs for the optimiser sweep that proposes new aggregates on this model:

  • Cron — when the optimiser runs.
  • Lookback hours — how far back the optimiser reads telemetry.
  • Max creates per run and auto-create cap per sweep — upper bounds.
  • Confidence threshold — recommendations below this are dropped.

Pocket

Pocket-table behaviour for this model: enabled toggle, auto-create, max rows, hit thresholds, benefit score, distinct ratio, TTL.

Predictive

Predictive-aggregate generation: evaluation window, validation minimum hits, unused retire days.

Limits

Per-model ceilings (max rows per query, requests per minute). All limits default to "no cap"; leaving the field blank means no model-specific limit is enforced.

Inheritance preview

Each row's caption reads Inherits: <value> and shows the value the resolver would return if the override were cleared. When you save a value, the row gets an "Overridden" chip. Clearing returns the row to the inherited state immediately.

Best practices

  • Override late. Start with project-level defaults. Only push a setting to the model level when telemetry shows the model behaves differently from its peers.
  • Document why. Leave a note in the model's description so the next modeller does not "tidy it up".
  • Watch for drift. If three or four models carry the same override, lift it to the project or system level.

Common pitfalls

  • Treating limits as recommendations. A blank limit means no cap, not "use the system default".
  • Setting an aggregate cron that conflicts with the optimiser. Stagger them.
  • Tuning the confidence threshold without a baseline. Wait for a week of usage data.
