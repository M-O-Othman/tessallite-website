Data Quality Rules

What this covers

Data quality rules let you declare constraints on physical source columns and automatically detect violations after each aggregate refresh. This page explains the five rule types, when to use them, how violations are recorded, and the block_on_failure option.

Why data quality rules matter

Aggregates are only useful if the underlying source data is clean. A fact table with NULL transaction amounts produces misleading subtotals. Duplicate keys on a dimension corrupt join cardinalities. A date column accepting free text silently breaks time-variant measure calculations.

Data quality rules close this loop. They run automatically after each refresh and write violation counts back to the aggregate metadata. When violations breach a threshold, a model alert is raised. If block_on_failure is enabled, the alert is escalated to error severity and clearly signals that downstream data should not be trusted.

Rule types

Rule typeWhat it checksConfiguration
not_nullCount of NULL values in the columnNo extra config
uniqueCount of duplicate values in the columnNo extra config
rangeRows where the value falls outside [min, max]{"min": 0, "max": 100}
regexRows where the column does not match the pattern{"pattern": "^\\d{4}-\\d{2}-\\d{2}$"}
custom_sqlHand-written SQL returning violation count as first column{"sql": "SELECT COUNT(*) FROM t WHERE ..."}

custom_sql considerations

  • Runs against the source database, not against the aggregate.
  • Must return a single numeric value in the first column of the first row.
  • Return 0 for “no violations”, a positive integer for N violations.
  • Parameterless SQL only. Session state, temp tables, and DDL are not supported.

Targets

target_typeMeaning
columnA specific ModelColumn; target_id is the column UUID
dimensionThe source column of a dimension; target_id is the dimension UUID
measureThe source column of a standard measure; target_id is the measure UUID

Severity levels

SeverityEffect
infoNo model alert raised
warnWarning alert raised
errorError alert raised

block_on_failure

When block_on_failure is enabled on a rule:

  • Any violation count > 0 produces an error-severity model alert, regardless of the rule’s severity setting.
  • The alert detail reads “Queries blocked until resolved.”
  • The aggregate’s violation chip in the Aggregates panel shows an error badge.

Use block_on_failure for rules where a single violation means the data is definitively wrong. Examples: a primary key column that must be unique, a date column that must never be NULL.

Automatic validation

Rules run automatically after each full aggregate refresh. You can also trigger validation manually using the Run Checks button in the Data Quality panel.

After validation:

  • last_violation_count on each rule is updated.
  • New violation history rows are written with timestamp, count, and a sample of failing values.
  • Model alerts are raised for rules that have violations and severity ≠ info.

Violation history

Click the expand button on a rule row to see past violation events. To clear stale history, click Clear on the rule row. Clearing sets last_violation_count to null and removes all violation history for that rule.

Common patterns

Fact table NOT NULL guard

Create a not_null rule targeting the primary key column with severity=error and block_on_failure=true. Any NULL means the CTAS joined incorrectly.

Dimension uniqueness guard

Create a unique rule targeting the natural key of a dimension. A duplicate key means two rows will join to the same fact row, inflating measures.

Date format guard

For a string-typed date column, create a regex rule with a date pattern. Out-of-format values cause time-variant measure calculations to fail silently.

Business range check

For a unit_price column, create a range rule with min=0. Negative prices indicate upstream ETL issues.

Limits

  • Rules run synchronously after each refresh. Many rules on a large fact table extend refresh time.
  • custom_sql rules run against the source database using the model’s source connection.
  • Deleting a rule also deletes its violation history.
