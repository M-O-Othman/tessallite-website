Cross-model calculation recipes

What this covers

A recipe is a multi-step calculation the agent can run against more than one model in a single turn. Recipes solve the class of analytical questions whose answer requires combining numbers from different fact tables. This article explains when a recipe is the right tool, how the recipe spec is shaped, how the engine executes it, and how to author recipes that the agent picks up reliably.

Why recipes exist

The semantic engine answers single-model questions: one fact table, one set of measures and dimensions, one time window. Some questions are inherently multi-model — a ratio across facts, a funnel across facts, a re-baselining. Without a recipe the LLM would have to ask each query separately and stitch the results in narration; in practice that path is unreliable. A recipe gives the engine a deterministic plan to execute and the LLM a single result to narrate.

What a recipe is

A recipe is a small JSON document a modeller authors. It has: a name and description (read by the answer LLM), parameters the user's question fills in, an ordered list of single-model steps, an optional combine expression, and free-form notes the judge reads as a hint about good output. Recipes live under Tenant Administration → Projects → <project> → Conversational agent → Recipes.

When to write a recipe

Write a recipe when the answer requires data from two or more facts that share at least one dimension, the combine is stable, and users ask the question often enough to merit a single-shot answer. Do not write a recipe when a single-model query already answers the question, when the combine logic is bespoke per question, or when dimensions diverge across facts at incompatible grains.

Anatomy of a recipe

A working example computes inventory turnover by dividing COGS by average inventory.

{
  "name": "Inventory turnover",
  "description": "Inventory turnover is COGS divided by average inventory over the same window.",
  "parameters": [
    { "name": "window", "description": "Time window.", "resolves_to_glossary_entity": true }
  ],
  "steps": [
    { "name": "cogs", "model_id": "...", "measures": ["cogs"], "dimensions": [],
      "filters": [{"name": "order_date", "op": "in_window", "value": "{window}"}], "limit": 1 },
    { "name": "avg_inventory", "model_id": "...", "measures": ["avg_inventory"], "dimensions": [],
      "filters": [{"name": "snapshot_date", "op": "in_window", "value": "{window}"}], "limit": 1 }
  ],
  "combine": "cogs / avg_inventory",
  "notes": "Express as a unitless ratio with two decimal places. Mention the window."
}

How the LLM picks a recipe

The answer LLM is shown the description and parameters of every recipe in the project. When the user's question matches a recipe's description well, the LLM emits a run_recipe tool call instead of a query tool call. Two practical implications: descriptions matter (write them in the words your users use), and overlapping recipes confuse the LLM (merge or sharpen them).

Common pitfalls

  • Mismatched grains. Daily and monthly facts combined without re-grain produce numbers that look right but are off by an order of magnitude.
  • Hard-coded time windows. Use a parameter instead.
  • Combines that hide divisions by zero. Add a note in the recipe so narration handles it.
  • Recipes that wrap a single query. Use a query.
  • Forgetting to allow-list both models. A recipe referencing a non-allow-listed model refuses at execution time.

Operating recipes

After authoring, exercise the recipe from Chat, watch the judge verdict on Recent calibration, and add the question to the per-model glossary's Example questions so the eval runner exercises the recipe path on every run.
