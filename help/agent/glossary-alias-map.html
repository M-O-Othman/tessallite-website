Author the glossary alias map

What this covers

The glossary alias map is the per-model context the agent reads alongside the project brief. It teaches the answer LLM how a model is shaped, what its measures actually mean, and how to disambiguate the abbreviations users type. This article explains the parts of the glossary, why each part matters for answer quality, and how to write entries that hold up under real questions.

Why a glossary

An LLM is a pattern matcher; without grounding it picks the most-likely-looking measure even when wrong. Every business invents its own shorthand — gross, NMV, take rate — and analysts type that shorthand into the chat box. The glossary closes both gaps. For each allow-listed model, the modeller writes a short overview, lists the analytical questions the model is good at, and supplies a disambiguation table for the abbreviations real users send.

The four parts

Each glossary entry is per-model. Open Project Agent → Models and pick a model to edit.

1. Model overview

Two to four sentences describing what the model is. Avoid cataloguing measures here — the engine already exposes those. Describe the grain, the time window, and the use case. A good overview answers "what is this model for, and what is it not for".

2. Analytical capabilities

Free-form prose. List the kinds of questions this model can answer well. Be specific about scope: a sentence noting what the model cannot answer is as valuable as listing what it can.

3. Abbreviation conflict rules

The most directly load-bearing field. List each business term that is ambiguous in your context, and the rule for resolving it. Two patterns work well — defaults ("By default, revenue means net revenue") and discriminators ("When the user says gross, prefer the gross-revenue measure"). If a term is unambiguous, do not add a rule for it.

4. Example questions

Five to ten worked examples. Each entry pairs a question with a short note on how it should decompose: which measure, which dimension, which time window. The agent uses these as in-context exemplars; the eval runner uses them as a regression suite.

Auto-derived sections

Three sections are populated automatically and surface in the same panel: an aggregates summary, a calendar alias list, and a dimension alias list. These are read-only — they are derived from the published model. Click Refresh derived after every meaningful model edit.

How the LLM uses the glossary

When a question lands, the prompt assembler concatenates: the project brief, then for each allow-listed model the four glossary parts plus the auto-derived sections, then the tool spec, then any conversation history, then the user message. The LLM emits a tool call. A long, accurate glossary noticeably reduces clarify and refuse rates.

Common pitfalls

  • Long overviews. A six-paragraph overview crowds out the example questions. Two to four sentences is the right length.
  • Generic example questions. "Show me the data" teaches nothing. Each example should name a measure, a dimension, and a time window.
  • Abbreviation rules for unambiguous terms. They cost tokens and add no value.
  • Not refreshing derived sections after a model change. The auto-derived calendar/dimension lists are a snapshot.
  • Forgetting cross-model rules. If two models share a term that means different things, write the rule on both model glossaries.
