Upgrade

What this covers

How to upgrade a running Tessallite installation to a new version for both local (Docker Compose) and GCP deployments. Includes database backup, image update, health verification, and rollback procedure.

Always back up the internal PostgreSQL database before upgrading. A new version may apply schema migrations that cannot be reversed automatically.

Local upgrade (Docker Compose)

  1. Back up the internal database. 
    docker compose exec postgres pg_dump -U tessallite tessallite > backup.sql
    Store backup.sql in a location outside the Docker volume before continuing.
  2. Pull the new images. 
    docker compose pull
    This downloads updated images for all services without stopping the running containers.
  3. Apply the new containers. 
    docker compose up -d
    Docker Compose restarts each service with the new image. Services are restarted one at a time in dependency order. Brief gaps in availability may occur for individual services; gateway clients may need to reconnect.
  4. Verify service health. 
    docker compose ps
    All services should show status Up. If any service shows Exit or Restarting, check its logs: 
    docker compose logs -f SERVICE_NAME
  5. Verify the database migration version. 
    curl http://localhost:3000/api/health
    The response includes a db_migration field. Compare it to the expected migration version listed in the release notes for the new version.

GCP upgrade (Cloud Run)

  1. Back up Cloud SQL. Create a manual backup before deploying new images: 
    gcloud sql backups create --instance=INSTANCE_NAME
    Verify the backup appears in the Cloud SQL backups list before continuing.
  2. Update each service image. For each Cloud Run service, deploy the new image tag: 
    gcloud run services update SERVICE_NAME \
  --image IMAGE_REPOSITORY/SERVICE_NAME:NEW_TAG \
  --region REGION
    Cloud Run performs a zero-downtime rolling replacement: it starts instances of the new revision before terminating instances of the old revision. Repeat for each service: model-service first, then query-router, optimizer, scheduler, frontend, gateway.
  3. Verify each service. Check the health endpoint for each service after its update: 
    curl https://SERVICE_URL/api/health
    Confirm {"status":"ok"} is returned before updating the next service.
  4. Verify the migration version on the frontend service: 
    curl https://FRONTEND_URL/api/health
    Confirm the db_migration field matches the expected version in the release notes.

Rollback

If a service fails after upgrade:

  • Local: Edit docker-compose.yml to pin the previous image tag, run docker compose up -d, then restore the database: 
    docker compose exec -T postgres psql -U tessallite tessallite < backup.sql
  • GCP: Roll back the Cloud Run revision to the previous version from the Cloud Console (Revisions tab > select previous revision > Send all traffic). Then restore Cloud SQL from the manual backup created in step 1.
